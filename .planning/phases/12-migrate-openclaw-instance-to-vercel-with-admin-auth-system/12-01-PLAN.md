    **Then update middleware.ts:**
    1. Add /admin/* protection BEFORE existing dashboard checks:
    ```typescript
    // Protect /admin routes — require admin role
    if (request.nextUrl.pathname.startsWith('/admin')) {
      if (!user) {
        const url = request.nextUrl.clone()
        url.pathname = '/login'
        return NextResponse.redirect(url)
      }
      const role = user.user_metadata?.role
      if (role !== 'admin') {
        const url = request.nextUrl.clone()
        url.pathname = '/'
        return NextResponse.redirect(url)  // Silently redirect non-admins
      }
    }
    ```

    2. Change the authenticated user login/signup redirect from `/dashboard` to `/`:

    // Line ~54: Change url.pathname = '/dashboard' to:
    url.pathname = '/'

    3. Keep existing /dashboard, /missions, /analytics, /settings protection for now (removed in Plan 02) — unauthenticated users still redirected to /login for those routes.

    **One-time admin user setup (CRITICAL):**
    After implementing the middleware, the admin Supabase user MUST have `user_metadata.role='admin'` set. Without this, the middleware will redirect even the intended admin to `/`. Set this via Supabase Dashboard: Authentication > Users > select admin user > Edit User > Raw user_metadata > add `{"role": "admin"}`. Alternatively, run via Supabase Admin API:
    ```typescript
    const { error } = await supabaseAdmin.auth.admin.updateUserById(ADMIN_USER_ID, {
      user_metadata: { role: 'admin' }
    })
    ```
    This is a one-time manual step — document it in the SUMMARY for Plan 03's human-verify checkpoint.

    Security note: Using user_metadata.role per locked decision. user_metadata CAN be modified by the user themselves — consider migrating to app_metadata in a future hardening phase.
