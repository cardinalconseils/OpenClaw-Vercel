## Summary

Phase 1.1 installs the OpenClaw agent framework (version 2026.3.13, the current latest), wires dual LLM providers, defines the Murphy persona through OpenClaw's workspace file system, installs and configures the `@openclaw/voice-call` plugin for Telnyx, and verifies the end-to-end pipeline.

OpenClaw stores its configuration in `~/.openclaw/openclaw.json` (JSON5 format) and uses a workspace directory (`~/.openclaw/workspace/`) for markdown persona files (SOUL.md, IDENTITY.md, AGENTS.md). The gateway runs as `openclaw start` on `ws://127.0.0.1:18789` — already wired in the existing GatewayManager. The voice-call plugin is a first-party OpenClaw plugin that handles Telnyx Call Control v2 webhooks internally, routing inbound call audio through an STT→LLM→TTS pipeline.

The key insight is that Phase 1.1 works at the **configuration layer**, not the code layer: write `openclaw.json`, write workspace persona files, install the voice-call plugin, and verify via the existing gateway. The Express webhook handler in `webhooks.ts` currently stubs events with `setImmediate()` — Phase 1.1 connects this to the OpenClaw gateway via WebSocket RPC (`sessions_send`), while the voice-call plugin handles the Telnyx call audio path independently.

**Primary recommendation:** Configure openclaw.json with tiered model routing (Gemini 2.5 Flash Lite via OpenRouter for routine, Claude Sonnet 4.5 via Anthropic for complex), write Murphy's persona into workspace markdown files, install the voice-call plugin, then verify with `openclaw agent --message "..."` for the text path and a simulated Telnyx webhook POST for the call path.

### Pattern 1: Tiered LLM Router

**What:** A single `chat()` function that routes to cheap (OpenRouter) or capable (Anthropic) based on task classification.

**When to use:** All LLM calls go through this router — never call providers directly from business logic.

**Example:**
```typescript
// src/lib/ai/orchestrator.ts
type LLMTier = 'routine' | 'complex';

const ROUTINE_TASKS = ['greeting', 'intent-capture', 'filler', 'status-update'];
const COMPLEX_TASKS = ['disambiguation', 'provider-ranking', 'transfer-logic'];

export async function chat(messages: Message[], task: string): Promise<string> {
  const tier: LLMTier = ROUTINE_TASKS.includes(task) ? 'routine' : 'complex';

  if (tier === 'routine') {
    // OpenRouter: google/gemini-2.5-flash-lite — low latency, cheap, good function calling
    return openRouterClient.chat.completions.create({ model: 'google/gemini-2.5-flash-lite', messages });
  } else {
    // Anthropic: claude-sonnet-4-5 — reasoning, disambiguation, ranking
    return anthropicClient.messages.create({ model: 'claude-sonnet-4-5-20251022', messages });
  }
}
```

### Pattern 4: Voice-Call Plugin Configuration in openclaw.json

**What:** The built-in `@openclaw/voice-call` plugin handles the Telnyx audio path. It runs INSIDE the gateway process.

**Important constraint:** The voice-call plugin starts its OWN webhook server on a separate port (default 3334). This conflicts with the existing Express server on 18790 receiving Telnyx webhooks. The architecture decision for Phase 1.1: **use the plugin's built-in webhook server** for call audio events, and route call-level JSON events (non-audio) through Express on 18790. This means two webhook URLs in Telnyx — one for call control (Express/18790), one for voice streaming (plugin/3334).

```json5
// ~/.openclaw/openclaw.json (JSON5 format — comments allowed)
{
  // Gateway settings
  gateway: {
    bind: "loopback",  // SECURITY: never 0.0.0.0
    port: 18789,
    token: process.env.OPENCLAW_GATEWAY_TOKEN,
  },

  // Agent defaults — coordinator/worker pattern
  agents: {
    defaults: {
      model: {
        primary: "openrouter/google/gemini-2.5-flash-lite",  // routine tasks
        fallbacks: ["anthropic/claude-sonnet-4-5"],           // complex tasks
      },
    },
    named: {
      murphy: {
        model: {
          primary: "openrouter/google/gemini-2.5-flash-lite",
          fallbacks: ["anthropic/claude-sonnet-4-5"],
        },
        systemPromptFile: "workspace/murphy-system.md",
      },
    },
  },

  // Auth profiles for each provider
  auth: {
    profiles: {
      openrouter: {
        mode: "api_key",
        key: process.env.OPENROUTER_API_KEY,
      },
      anthropic: {
        mode: "api_key",
        key: process.env.ANTHROPIC_API_KEY,
      },
    },
  },

  // Voice-call plugin — Telnyx provider
  plugins: {
    entries: {
      "voice-call": {
        enabled: true,
        config: {
          provider: "telnyx",
          fromNumber: process.env.TELNYX_PHONE_NUMBER,
          inboundPolicy: "open",  // Phase 1.1: open for testing; tighten in later phases
          telnyx: {
            apiKey: process.env.TELNYX_API_KEY,
            connectionId: process.env.TELNYX_CONNECTION_ID,
            publicKey: process.env.TELNYX_PUBLIC_KEY,
          },
          serve: {
            port: 3334,
            path: "/voice/telnyx",
          },
        },
      },
    },
  },
}
```

### Anti-Patterns to Avoid

- **Calling LLM providers directly from webhooks.ts:** Always go through the orchestrator/router — enables tier switching without touching business logic.
- **Putting persona in code strings:** Murphy's personality belongs in SOUL.md, not in a TypeScript const. SOUL.md is hot-reloadable, diffable, and version-controlled independently.
- **Using `express.json()` globally for the agent route:** Already established pattern — raw body parser at route level only. Do not break this for any new routes.
- **Wiring the voice-call plugin's webhook URL to the same Express port (18790):** The plugin runs its own server on 3334. Configure Telnyx with both URLs, not one.
- **`inboundPolicy: "allowlist"` with empty `allowFrom` in Phase 1.1:** Blocks all calls. Use `"open"` for development; allowlist in production.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| STT (Speech to Text) | Custom Deepgram/Whisper integration | `@openclaw/voice-call` plugin | Plugin handles PCM audio, streaming, Telnyx media streams, codec negotiation |
| TTS (Text to Speech) | Custom ElevenLabs/OpenAI TTS wiring | Plugin's `messages.tts` config | Plugin handles voice format (PCM required for telephony — Edge TTS explicitly not supported) |
| Webhook audio server | Second Express app for voice audio | Plugin's built-in `serve.port: 3334` | Plugin serves its own webhook endpoint with Telnyx signature verification built in |
| Model failover/retry | Custom try/catch with fallback | OpenClaw native fallbacks array in `openclaw.json` | Gateway handles failover, rate limits, and auth profile rotation |
| Gateway session management | Custom WebSocket session tracking | OpenClaw's session system (`voice:{phone}` keys) | Sessions are persisted as JSONL transcripts automatically |
| AI identity compliance disclosure | Custom logic to inject disclaimer | System prompt + `SOUL.md` rule | Simpler to enforce at persona level than at code level |

**Key insight:** OpenClaw's voice-call plugin is a complete STT→LLM→TTS pipeline. The application's job is to configure it correctly, not to implement any audio processing.

## Common Pitfalls

### Pitfall 1: Node Version Requirement

**What goes wrong:** `openclaw start` fails silently or produces cryptic errors.
**Why it happens:** OpenClaw requires Node >= 22. The project uses Node 20+. The `package.json` `engines` field may not match.
**How to avoid:** Run `node --version` in the Vercel Sandbox environment. If it's Node 20, update the runtime or use the `--experimental-vm-modules` flag. Check OpenClaw npm page: stated requirement is Node >= 22.
**Warning signs:** `openclaw` CLI exits immediately with no output; `openclaw doctor` fails at Node version check.

### Pitfall 2: openclaw.json Silently Fails Validation

**What goes wrong:** Gateway starts but ignores parts of the config (no error).
**Why it happens:** openclaw.json uses Zod validation with hot reload — unknown keys or invalid types prevent Gateway boot, but the error may only appear in gateway stderr, not in the spawned process stdout piped through GatewayManager.
**How to avoid:** Run `openclaw doctor` after editing config. Check `[gateway:err]` prefixed logs in the existing GatewayManager stderr handler.
**Warning signs:** Gateway is healthy (200 on port 18789) but named agent "murphy" is not listed in `openclaw status`.

### Pitfall 3: Voice-Call Plugin Port Conflict

**What goes wrong:** Plugin fails to start because port 3334 is occupied, or Telnyx events go to wrong port.
**Why it happens:** Plugin starts its own HTTP server inside the gateway process. If anything else binds 3334 first, it fails.
**How to avoid:** Use a non-conflicting port for the voice plugin (3334 is the default — safe if nothing else uses it in the Sandbox). Add port to `sandbox-start.sh` health check.
**Warning signs:** Voice events reach Express on 18790 but agent doesn't respond; `openclaw voicecall status` shows plugin not ready.

### Pitfall 4: Telnyx Webhook Public Key Format

**What goes wrong:** Telnyx signature verification fails with "invalid public key" error.
**Why it happens:** The voice-call plugin's `telnyx.publicKey` must be the Base64-encoded Ed25519 public key from the Telnyx Mission Control Portal, NOT the raw key bytes. The existing `webhooks.ts` already handles this correctly via `telnyxClient.webhooks.unwrap()` — but the plugin needs the same key in a different config format.
**How to avoid:** Copy the key from Telnyx portal as-is (it's already Base64). Set `TELNYX_PUBLIC_KEY` env var.
**Warning signs:** Plugin logs "webhook signature mismatch"; inbound call events never reach the agent.

### Pitfall 5: Voice Session Isolation

**What goes wrong:** Murphy can't remember what was discussed on a call.
**Why it happens:** The voice-call plugin creates isolated `voice:{phone}` sessions — separate from any text/RPC sessions. This is a known OpenClaw limitation (Issue #17592, closed as "not planned").
**How to avoid:** Accept the isolation for Phase 1.1 (verification only). In Phase 2, bridge context using `sessions_send` RPC to copy relevant context from voice session to main session.
**Warning signs:** Murphy greets the caller as if it's their first call even when they called before.

### Pitfall 6: OpenRouter Model ID Format

**What goes wrong:** Model not found error or fallback to wrong model.
**Why it happens:** OpenRouter model IDs use `author/slug` format (e.g., `google/gemini-2.5-flash-lite`), but in openclaw.json they must be prefixed: `openrouter/google/gemini-2.5-flash-lite`.
**How to avoid:** Use the `openrouter/` prefix in openclaw.json for any OpenRouter model. Verify with `openclaw models list`.
**Warning signs:** Routine tasks unexpectedly route to Anthropic (expensive); `openclaw doctor` warns about unknown model.

### Pitfall 7: AI Disclosure Compliance

**What goes wrong:** Legal exposure if Murphy greets without disclosing AI nature.
**Why it happens:** FCC requires disclosure at the beginning of each AI-voice call. CA SB-1001 (online bots) and the FCC's TCPA interpretation both require clear disclosure before substantive conversation.
**How to avoid:** Murphy's greeting MUST include disclosure before any service-related content. Recommended wording: "Hi, I'm Murphy — an AI assistant from OpenClaw. I help find and connect you with local service providers." The disclosure ("AI assistant") must be in the first sentence, clearly audible, not buried.
**Warning signs:** None at runtime — this is a compliance risk, not a technical one.

### Tiered LLM Orchestrator

```typescript
// src/lib/ai/orchestrator.ts
import { openRouterClient, anthropicClient } from './llm-clients.js';

type TaskType = 'greeting' | 'intent-capture' | 'filler' | 'status-update'
              | 'disambiguation' | 'provider-ranking' | 'transfer-logic' | 'unknown';

const ROUTINE_TASKS: TaskType[] = ['greeting', 'intent-capture', 'filler', 'status-update'];

export async function chat(
  messages: Array<{ role: 'user' | 'assistant'; content: string }>,
  task: TaskType = 'unknown'
): Promise<string> {
  const isRoutine = ROUTINE_TASKS.includes(task);

  if (isRoutine) {
    // Cheap, fast path — OpenRouter / Gemini 2.5 Flash Lite
    const response = await openRouterClient.chat.completions.create({
      model: 'google/gemini-2.5-flash-lite',
      messages,
      max_tokens: 200,  // Voice responses are short
    });
    return response.choices[0]?.message?.content ?? '';
  } else {
    // Capable path — Anthropic Claude Sonnet 4.5
    const response = await anthropicClient.messages.create({
      model: 'claude-sonnet-4-5-20251022',
      max_tokens: 500,
      messages,
    });
    const block = response.content[0];
    return block?.type === 'text' ? block.text : '';
  }
}
```

## Open Questions

1. **Node >=22 requirement vs Vercel Sandbox Node version**
   - What we know: OpenClaw npm page states Node >=22; Vercel Sandbox environment ships Node 20
   - What's unclear: Whether the Sandbox has been updated to Node 22, or whether there's a workaround
   - Recommendation: Wave 0 — run `node --version` in Sandbox before any agent code. If Node 20, either request Sandbox upgrade or check if openclaw 2026.3.13 actually tolerates Node 20 (some npm `engines` fields are advisory-only)

2. **OpenRouter API key — not in .env.example**
   - What we know: `.env.example` has `ANTHROPIC_API_KEY` but no `OPENROUTER_API_KEY`
   - What's unclear: Whether an OpenRouter account exists or if the team is using a different routing approach
   - Recommendation: Add `OPENROUTER_API_KEY=` to `.env.example` and document that it's required for Phase 1.1

3. **Voice-call plugin webhook URL vs Express webhook URL**
   - What we know: Plugin uses its own server on port 3334; Express uses 18790; Telnyx supports multiple webhook URLs per application
   - What's unclear: Whether the Vercel Sandbox can expose BOTH ports publicly (may need two separate ngrok/tunnel configurations)
   - Recommendation: Investigate Sandbox port exposure in Wave 0. If only one public port is possible, route plugin webhook through Express (requires custom proxy middleware)

4. **`openclaw.json` location in Vercel Sandbox**
   - What we know: Default location is `~/.openclaw/openclaw.json` (user home directory)
   - What's unclear: Whether the Sandbox home directory persists between invocations, or if config must be written at startup
   - Recommendation: Add `sandbox-start.sh` step to write `openclaw.json` from environment variables before `openclaw start`
